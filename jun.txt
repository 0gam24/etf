## 2026-04-25 · [Cloudflare 일반]

### Q. Cloudflare는 한 사이트밖에 못 해?
A. 아니요, 무제한입니다.
- Cloudflare Free 플랜에서 도메인(사이트) 수 제한 없음
- DNS 레코드 도메인당 무제한
- Workers 무료 100개 (일 10만 요청)
- Pages 프로젝트 무제한
- SSL 인증서 모든 도메인 자동 무료
호스팅케이알에 도메인 두고 CNAME만 가리키는 대안도 있지만 루트 도메인 불가·SSL 미자동·CDN 혜택 없음. Cloudflare로 옮기는 게 정답.


## 2026-04-25 · [멀티 프로젝트 운영]

### Q. 다른 웹사이트도 같은 방식으로 만들 거야. GitHub·Cloudflare 여러 프로젝트 가능?
A. 둘 다 무제한.
- GitHub Public Repo: 무제한 (Actions도 무제한)
- GitHub Private Repo: 무제한 (Actions 월 2,000분 공유)
- Cloudflare Workers: 100개, Pages 무제한
- 도메인 등록은 호스팅케이알이나 Cloudflare Registrar(원가 $9.15/년)
- 새 프로젝트마다: 별도 repo + 별도 Worker + 별도 도메인(또는 서브도메인)
- 효율 팁:
  - Gemini API 키는 프로젝트별 분리 권장 (각 일 20건 한도 별도 확보)
  - 이번 etf 프로젝트 완성 후 fork·복제로 새 콘텐츠 사이트 빠르게 생성 가능
  - Cloudflare 한 계정에 여러 도메인 묶어 통합 관리

## 2026-04-25 · [배포 자동화 안전성]

### Q. 앞으로 프론트엔드/기획을 계속 수정 보완할 건데, 그때마다 빌드 → 커밋 → push → 자동 재배포 사이클 문제 없을까?
A. 대부분 무문제. 5가지 주의 + 권장 워크플로 정리:

**한도 (Free 플랜)**
- Cloudflare Workers 빌드: 월 500회 (일 10번 push해도 월 300회 → 충분)
- Workers 요청: 일 10만건
- 번들 크기: 1MB (현재 ~600KB 여유)

**5가지 리스크 + 대응**
1. 잘못된 코드 push = 즉시 라이브 → Cloudflare 1-클릭 롤백 가능 (Deployments → 직전 버전 → Rollback)
2. local push와 GitHub Actions cron 동시 = 충돌 → push 전 `git pull --rebase` 습관
3. preview 없이 main 직접 push = 위험 → 큰 변경은 feature 브랜치 + Cloudflare 자동 preview URL
4. 빌드 횟수 급증 = 캐시 미스 → 작은 수정 묶어서 한 번에 push
5. 콘텐츠+코드 git log 섞임 → grep로 분리 가능, 실 영향 없음

**권장 워크플로 (변경 크기별)**
- 작은 수정 (텍스트·CSS): main 직접 push (현재 방식)
- 중간 (라우팅·UI 개편): feature 브랜치 → PR → preview URL → merge
- 큰 (에이전트·파이프라인): 위 + 로컬 `npm run cf:build` + `npm run pulse` 사전 검증

**안전망**
- Cloudflare 자동 preview URL (모든 브랜치)
- 1-클릭 롤백 (5초)
- 로컬 사전 검증 3종: dev / cf:build / pulse


## 2026-04-25 · [배포 흐름 개념]

### Q. 로컬에서 수정 → GitHub push → Cloudflare로 가는 게 맞아? 개념부터 잡아야 해.
A. 정확히 그 흐름.

```
[로컬 PC]              [GitHub main]              [Cloudflare]
 코드 수정 ── push ──▶  새 commit ── webhook ──▶ 자동 빌드 + 배포
                                                   ↓
                                               iknowhowinfo.com
                                               (3~5분 후 라이브)
```

**단계 상세**
1. 로컬에서 파일 수정 (VSCode)
2. git add → git commit → git push (1초+30초)
3. Cloudflare가 GitHub webhook으로 push 감지 (1초)
4. Cloudflare가 최신 코드 clone + npm run cf:build (2~3분)
5. Workers에 배포 + iknowhowinfo.com 새 버전 적용 (30초)
6. 총 3~5분 후 브라우저에서 확인 가능

**두 번째 흐름 — 매일 09:00 자동 cron**
- GitHub Actions 서버에서 cron 실행 (사용자 PC 안 켜져도 동작)
- npm run pulse → 새 MDX·데이터 생성 → 봇이 git push
- Cloudflare가 같은 webhook으로 자동 빌드·배포

**핵심 개념**
- 모든 길은 GitHub main을 통과
- Cloudflare는 항상 GitHub의 main을 source-of-truth로 봄
- 로컬 → GitHub → Cloudflare 한 방향만 흐름
- Cloudflare에 직접 코드 업로드 불필요

**자주 헷갈리는 점**
- Q: 로컬과 GitHub이 다르면? → 매일 cron이 push하므로 작업 시작 전 항상 `git pull --rebase`
- Q: 커밋 안 하고 push만? → 불가능, git push는 commit된 변경만 전송



## 2026-04-25 · [Git 기초]

### Q. push와 commit 차이가 뭐야?
A. 저장 위치가 다른 두 단계.

**한 줄 요약**
- commit = 내 PC 안 저장소(.git)에 스냅샷 박기 — 인터넷 없어도 가능, 나만 봄
- push = 그 스냅샷을 GitHub 서버로 업로드 — 인터넷 필요, 공개됨, Cloudflare 자동 배포 트리거

**흐름**
```
파일 수정 ─add─▶ staging ─commit─▶ 로컬 .git ─push─▶ GitHub ─webhook─▶ Cloudflare
```

**비유 (책 출판)**
- commit = 챕터 끝나면 USB에 백업 (내 책상 안)
- push = 출판사에 원고 보내기

**왜 둘로 나뉘었나**
- 오프라인 작업 가능 (commit 누적 후 한꺼번에 push)
- 실수 방지 (commit만 하면 GitHub은 깨끗)
- 효율성 (작은 commit 여러 개 → push 1번 → Cloudflare 빌드 1번)

**자주 묻는 패턴**
- commit만 하고 GitHub에 안 보임? → 정상, push 안 했으니까
- commit 없이 push만? → 불가능, push는 commit된 것만 보냄
- 여러 commit → 한 번에 push? → 일반적, Cloudflare는 마지막 push만 트리거
- push 후 잘못 발견? → 새 commit으로 수정(안전), git revert, 또는 Cloudflare 1-클릭 롤백

**우리 프로젝트 실전**
```bash
git pull --rebase    # 자동 cron이 push한 것 가져오기
# 파일 수정
git add -A
git commit -m "메시지"
git push             # 라이브 반영 트리거
```

### 보강 — "commit은 내 PC만, push는 GitHub로" 이렇게 외워도 돼?
A. 정확. push는 GitHub까지만 의식하면 됨. Cloudflare 빌드·배포는 GitHub webhook 받아 자동 동작 → 사용자 행동 불필요.
```
commit → 내 PC
push   → GitHub  → (자동) Cloudflare 빌드 → iknowhowinfo.com 라이브
```

## 2026-04-25 · [production 슬러그 404 근본 원인 + 수정]

### Q. 빌드 설정 확인했어 (Build command: npm run cf:build, Root: /, Branch: main).
A. 빌드 명령어는 정확. 진짜 원인은 **OpenNext의 `populateCache local` 단계 누락**. 수정 완료.

**진단 과정**
1. production sitemap 라인 수: 19개 (홈 1 + 카테고리 5 + 가이드 6 + 저자 7) — 글 슬러그 0개
2. → `getAllPosts()`가 빌드 시점에 동작하긴 했지만, prerender 결과물이 production에서 못 읽힘
3. 로컬 `.open-next/cache/{buildId}/pulse/pulse-20260425.cache` 파일은 정상 생성됨
4. 그러나 `.open-next/assets/cdn-cgi/_next_cache/` 폴더가 비어 있음 ← **결정적**
5. OpenNext Cloudflare 어댑터의 `static-assets-incremental-cache.js` 분석 → cache 파일은 `ASSETS.fetch('/cdn-cgi/_next_cache/{buildId}/{key}.cache')`로 읽음
6. `populate-cache.js#populateStaticAssetsIncrementalCache` 함수가 `.open-next/cache/` → `.open-next/assets/cdn-cgi/_next_cache/`로 cp하는데, `opennextjs-cloudflare build`는 이 함수를 자동 호출 안 함 → 별도 `populateCache` 커맨드로 호출 필요

**수정 (3줄)**
- `open-next.config.ts`: `staticAssetsIncrementalCache` import + `incrementalCache` 옵션 명시
- `package.json` cf:build: `opennextjs-cloudflare build && opennextjs-cloudflare populateCache local`
- `wrangler.jsonc` name: `etf-pulse` → `etf` (Cloudflare 대시보드와 일치, 경고 해소)

**검증**
- 로컬 rebuild → `.open-next/assets/cdn-cgi/_next_cache/{buildId}/pulse/pulse-20260425.cache` 등 18개 슬러그 캐시 파일 정상 복사
- 다음 push 시 Cloudflare가 새 빌드 트리거 → populateCache까지 실행 → wrangler deploy가 cache 폴더 포함한 assets 전체 업로드 → production에서 ASSETS.fetch()로 슬러그 cache 읽음 → SSG 슬러그 200 OK

**push 후 검증 명령어**
```
npm run verify:parity   # 18개 슬러그 모두 200 확인
```
